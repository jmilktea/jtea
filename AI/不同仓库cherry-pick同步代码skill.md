# 用 Cursor Skill 干掉每周最烦的半小时

## 一个真实的痛点

我们团队有一套微服务架构的系统，拆成了十来个 Git 仓库。由于合规要求，代码需要同时维护两个 Git 远端——日常开发在主远端推进，但每次发版都要把改动同步到内部仓库，两边一起发版。

同步方式是 cherry-pick——把开发分支上的 commit 逐个挑到内部仓库的对应分支上去。

这件事本身不难，但架不住重复。

## 手动操作到底有多烦

假设这周你在 `feature_login` 分支上提了 8 个 commit，涉及 3 个模块。同步的时候你要做什么？

先来第一个模块：

```bash
cd project-user
git fetch origin
git fetch internal
git log --oneline origin/master..feature_login    # 看看有哪些 commit 要搬
git checkout -b feature_login_internal internal/main
git cherry-pick origin/master..feature_login
git push internal feature_login_internal
```

好，第一个模块搞完了。切到第二个模块，把上面的命令再来一遍，仓库名换一下。然后第三个模块再来一遍。

这还是顺利的情况。不顺利的话——cherry-pick 到一半冲突了，解完冲突 `cherry-pick --continue`，发现下一个 commit 又冲突了。或者有几个 commit 之前已经同步过了，cherry-pick 报空 commit，你得判断是跳过还是真出了问题。

更头疼的是分支管理。全量同步还好说，增量同步的时候你得记住"上次同步到哪个 commit 了"，然后用 `commit_hash^..branch` 这种语法来指定范围。这东西你一周不碰就忘。

我粗略算过，如果涉及 5 个模块、每个模块平均 5-6 个 commit，手动操作一轮大概要 20-30 分钟。而且不能走神——一旦在错误的目录下执行了命令，或者忘了 fetch 就开始 cherry-pick，那恢复起来更费时间。

这活吧，说技术含量，没多少；说耗精力，真的耗。典型的"不值得人来干但又不敢不认真干"的事。

## 解决方案：用 Cursor Skill 自动化

后来我用 Cursor 的 Skill 机制把这套操作给自动化了。

Cursor Skill 本质上是一份 Markdown 格式的指令文档，告诉 AI 遇到特定请求时应该执行什么操作。它不是传统的脚本或 CI pipeline，而是一种"教 AI 干活"的方式——你把操作步骤、异常处理、交互逻辑全写清楚，AI 就能照着执行，同时保留和人交互的灵活性。

### 环境准备

用之前要做一点准备工作。Skill 文件需要放在工作区的 `.cursor/skills/` 目录下才能被 Cursor 识别。我们的做法是把 Skill 放在一个独立的仓库里维护，然后在项目根目录下创建一个符号链接指过去：

```
workspace/                          ← 用 Cursor 打开这个目录
├── module-a/                       ← 业务模块（各自独立的 Git 仓库）
├── module-b/
├── module-c/
├── ai-assistant/                   ← Skill 所在仓库（clone 到这里）
│   └── .cursor/skills/sync-internal/SKILL.md
└── .cursor → ai-assistant/.cursor  ← 符号链接
```

Windows 下用 `mklink /J` 创建目录联接：

```cmd
mklink /J "D:\workspace\.cursor" "D:\workspace\ai-assistant\.cursor"
```

另外每个业务模块需要已经配好两个远端（`origin` 和内部仓库的远端）。

### 怎么触发

环境就绪后，在 Cursor 的 AI 聊天框里输入触发词就行——我设的是"同步内部仓库"，你可以在 Skill 文件里改成自己习惯的说法。

**打一句"同步内部仓库"，AI 就开始跑了。**

### 整体流程

触发之后 AI 通过问答引导你走完全程：

1. **选分支和模块** —— AI 先问你从哪个分支同步（默认主分支，也可输入自定义分支名），再列出工作区里的所有业务模块让你多选勾选
2. **预览和选模式** —— 自动 fetch 两个远端，列出每个模块的新 commit，然后逐模块让你选 cherry-pick 方式：全量 / 增量 / 手动挑选。选完汇总展示计划，确认后才动手
3. **自动执行** —— AI 逐模块执行 cherry-pick 并推送。碰到空 commit 自动跳过，碰到真冲突暂停等你手动解决后回复"继续"
4. **汇总报告** —— 输出每个模块的状态、cherry-pick 数量、跳过数量和总耗时

整个过程从 20-30 分钟压缩到 3-5 分钟，其中大部分时间还是在等 git 操作跑完。我需要做的就是回答几个选择题，以及处理偶尔出现的冲突。

## Skill 里到底写了什么

Skill 文件本质是一份 Markdown，但不是随便写几句提示词就行。你得把整个操作流程、每一步的 git 命令、各种异常分支、和用户的交互方式全部写清楚。我这个 Skill 写了两百多行，拆开来看大概分四个阶段。

### 阶段一：收集参数

流程一开始，AI 用选择题的形式问两个问题：

1. **源分支**——从哪个分支取 commit。给两个选项：直接用主分支，或者手动输入分支名（比如 `feature_login`）。
2. **模块选择**——工作区里有哪些模块需要同步。AI 会自动扫描工作区目录，把所有业务模块列成多选框让你勾。

用选择题而不是让用户手敲，是为了避免拼错分支名或者漏选模块这种低级失误。

### 阶段二：预检和预览

用户选完之后，AI 对每个模块并行执行 `git fetch`，同时拉取两个远端的最新状态。然后逐模块做两件事：

**确定 commit 来源。** 这里有个设计细节——AI 会先检查本地有没有这个分支，有的话优先用本地分支作为 cherry-pick 的源。为什么？因为你可能有还没 push 到远端的 commit，这些 commit 只存在于本地分支上。如果本地没有，再 fallback 到 `origin/{分支名}`。如果两边都找不到这个分支，直接告诉你然后跳过这个模块。

判断逻辑用的是 `git rev-parse --verify`：

```bash
# 先查本地
git rev-parse --verify feature_login
# 本地没有，查远端
git rev-parse --verify origin/feature_login
```

**列出待同步的 commit。** 确定了来源之后，用 `git log` 列出主分支到源分支之间的差异：

```bash
git log --format="%h %ai %an %s" origin/master..{源引用}
```

所有模块的 commit 列表会汇总展示给你看，每个模块还会标注用的是本地分支还是远端分支。

接下来到了最关键的一步——逐模块选择 cherry-pick 模式。AI 会针对每个模块单独问你：

| 模式 | 含义 | 什么时候用 |
|------|------|-----------|
| 全量 cherry-pick | 从第一个 commit 到最新，全部搬过去 | 首次同步，或者想重建目标分支 |
| 增量 cherry-pick | 从你指定的某个 commit 开始，到最新 | 上次已经同步了一部分，这次只搬新增的 |
| 指定 cherry-pick | 手动勾选若干 commit，只搬选中的 | 只需要挑几个特定改动 |

选"增量"的话，AI 会把 commit 列表展示出来让你单选一个起始点；选"指定"则是多选勾选。这些交互全部通过 Cursor 的 AskQuestion 工具实现，不需要手动输入 hash 值。

最后 AI 把所有模块的计划汇总一遍——"模块 A 全量 8 个 commit，模块 B 从 `abc1234` 开始增量 3 个 commit，模块 C 指定 2 个 commit"——确认之后才开始执行。这一步确认很重要，相当于给你一个最后检查的机会。

### 阶段三：逐模块执行

确认后 AI 开始逐个模块跑。注意是**串行**，不是并行。并行效率高是高，但一旦有个模块冲突了，你分不清哪个模块在等你处理。串行的好处是出了问题能立刻定位。

每个模块的执行分三步：

**第一步：检查工作区。** 动手之前先看一眼当前目录有没有未提交的修改：

```bash
git status --porcelain --untracked-files=no
```

如果有——这个情况比想象中常见，你可能正在改另一个 bug，随手触发了同步——AI 会停下来问你：stash 暂存一下继续？跳过这个模块？还是整个流程先停了？

选 stash 的话 AI 会自动执行 `git stash push`，并且在最后的汇总报告里提醒你记得 `stash pop` 恢复。这个细节是我亲身经历了一次"切分支把改了一半的代码搞丢"之后加上去的。

**第二步：准备目标分支。** 根据你选的 cherry-pick 模式不同，分支操作也不一样：

- 全量模式：从内部仓库的基准分支 `main` 创建一条新的本地分支。如果同名分支已经存在，问你要不要删了重建。
- 增量/指定模式：checkout 到之前同步过的分支上，先 pull 一下最新代码。如果找不到之前的分支（本地和远端都没有），会问你要不要改成全量模式。

这段的分支名遵循固定规则——源分支名加个 `_internal` 后缀，比如 `feature_login` 对应 `feature_login_internal`。写死命名规则是为了不用每次人工记忆"上次那个分支叫啥来着"。

**第三步：执行 cherry-pick。** 这里是整个 Skill 最复杂的部分，因为要处理三种模式和两类异常。

三种模式对应不同的 git 命令：

```bash
# 全量：主分支到源分支之间的所有 commit
git cherry-pick origin/master..{源引用}

# 增量：从指定 commit（含）到最新
git cherry-pick {commit_hash}^..{源引用}

# 指定：按时间正序逐个 cherry-pick
git cherry-pick abc1234 def5678 ghi9012
```

cherry-pick 失败的时候（退出码非 0），需要区分两种情况。这个判断逻辑是整个 Skill 里我觉得最精巧的一段：

```bash
git diff --cached --quiet
```

执行这条命令，看退出码：

- **退出码 0**（暂存区没东西）→ 说明这是个空 commit，之前已经同步过了。自动 `cherry-pick --skip` 跳过，继续下一个。
- **退出码 1**（暂存区有变更）→ 说明是真冲突。AI 停下来，列出冲突文件清单，等你手动解决。

这个区分很关键。如果不做判断，一律当冲突处理，那每次碰到已同步的 commit 都要手动介入，烦不胜烦。但如果一律自动跳过，又可能把真冲突给吞了。用暂存区状态来判断是最靠谱的方式。

真冲突的处理流程也写得很明确：AI 告诉你哪些文件冲突了，然后**停下来等**。你用 IDE 或命令行解完冲突，回到聊天框说一句"继续"，AI 就执行 `git add . && git cherry-pick --continue` 接着往下走。如果下一个 commit 又冲突了，重复这个循环。

这里有条红线：**冲突时必须停下来让人处理**，这条写死在 Skill 里了。让 AI 自动解决代码冲突？涉及业务逻辑的东西，想都不要想。

cherry-pick 全部完成后，一条 push 推到内部远端：

```bash
git push internal {分支名}_internal
```

然后进入下一个模块，重复上述流程。

### 阶段四：汇总报告

所有模块处理完毕，AI 给一张这样的表：

| 模块 | 状态 | cherry-pick 数量 | 跳过（已同步） | 备注 |
|------|------|-----------------|---------------|------|
| project-user | 成功 | 3 | 0 | 已推送 |
| project-order | 有冲突已解决 | 5 | 2 | 跳过 2 个空 commit |
| project-common | 跳过 | 0 | 0 | 分支不存在 |

底下还带一行总耗时——从开始到结束花了多久。如果过程中有模块做了 stash，最后会额外提醒你恢复。

整个 Skill 从第一版写到现在，迭代了四五版。每一版的改进基本都是踩坑踩出来的：空 commit 判断是某次 cherry-pick 莫名报错之后加的；stash 检查是丢过一次代码之后加的；源引用优先级是发现本地未 push 的 commit 漏了之后加的。

## 一些写 Skill 的经验

写完这个 Skill 之后我也在想，什么样的操作适合用这种方式来自动化。总结下来有几条：

**步骤固定但细节多的操作最合适。** 这种 cherry-pick 同步流程，步骤是确定的（fetch → 切分支 → cherry-pick → push），但每一步的参数、异常分支不少。写成 shell 脚本也行，但处理交互和异常分支会很啰嗦。Skill 的好处是 AI 能理解语义，你写"如果冲突就停下来等用户处理"就够了，不需要写一堆 trap 和 read。

**需要人介入判断的环节别省。** 我最初想把冲突解决也交给 AI，后来果断放弃了。代码冲突涉及业务逻辑，让 AI 猜着解决迟早出事。该停就停，该让人看就让人看。

**异常路径要写得比正常路径更详细。** 正常流程 AI 其实很容易跑通，出问题的永远是边界情况。空 commit 怎么办、分支不存在怎么办、工作区不干净怎么办——这些才是 Skill 的真正价值所在。

**别一开始就追求完美。** 先把 happy path 跑通，实际用的时候遇到问题再补。我这个 Skill 迭代了四五版才到现在的样子。

## 写在最后

说到底这就是个"把经验写成文档，然后让 AI 照着干"的事。和写 Runbook 或者 SOP 差不多，只不过执行者从人变成了 AI。

它不酷也不炫，就是实实在在地帮我省了每周半小时的机械操作。对于这种"不难但烦"的重复性工作，我觉得是目前 AI 编程助手最务实的用法之一。
